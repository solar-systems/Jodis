## Write-Ahead Log (WAL)

### 概述
Write-Ahead Log（预写日志）是 Jodis 用于实现数据持久化和故障恢复的核心机制。所有对数据的修改操作都会先记录到 WAL 中，然后再应用到内存数据。

### WAL 的工作原理

1. **写入顺序**
   - 客户端发送写命令
   - 服务器先将命令追加到 WAL 文件
   - 然后执行命令并更新内存数据
   - 返回响应给客户端

2. **日志格式**
   - 每条日志记录包含：时间戳、操作类型、键、值等信息
   - 日志以追加方式写入，保证高性能

3. **故障恢复**
   - 服务器重启时，读取 WAL 文件
   - 重放所有记录的操作
   - 恢复到故障前的状态

### WAL 文件管理

#### 日志文件大小
- 默认单个 WAL 文件最大为 64MB
- 超过阈值后会自动创建新的日志文件

#### 日志重写（Rewrite）
- 当 WAL 文件过大时，触发重写机制
- 只保留当前数据的最新状态
- 丢弃已被持久化的旧日志

#### 日志合并
- 多个小的 WAL 文件可以合并为一个
- 减少文件数量，提高读取性能

### 配置文件参数
```properties
# WAL 日志文件名
log.wal=default.wal

# WAL 重写阈值（字节）
log.wal.rewrite.size=64 * 1024 * 1024

# 重载模式
# 0: 不加载
# 1: 仅加载 RDB
# 2: 加载 RDB + WAL（推荐）
log.reload.mode=2
```

### WAL vs RDB

| 特性 | WAL | RDB |
|------|-----|-----|
| **写入方式** | 追加写 | 全量快照 |
| **性能** | 高 | 较低 |
| **数据丢失风险** | 最小 | 可能丢失上次快照后的数据 |
| **文件大小** | 较大 | 较小 |
| **恢复速度** | 较慢 | 快 |

### 使用示例

#### 查看 WAL 配置
```bash
# 查看当前 WAL 文件大小
INFO persistence
```

#### 手动触发 WAL 重写
```bash
# 系统会自动在适当时机重写
# 无需手动干预
BGREWRITEAOF
```

### 最佳实践

1. **定期备份**
   - 定期备份 WAL 和 RDB 文件
   - 建议使用脚本自动化备份

2. **监控文件大小**
   - 监控 WAL 文件大小
   - 确保磁盘空间充足

3. **合理配置**
   - 根据业务需求调整 `log.wal.rewrite.size`
   - 生产环境建议设置 `log.reload.mode=2`

4. **故障处理**
   - 如果 WAL 文件损坏，可以尝试从 RDB 恢复
   - 删除损坏的 WAL 文件并重启服务器

### 注意事项

⚠️ **重要提示**：
1. WAL 文件不应手动编辑
2. 确保 WAL 目录有足够磁盘空间
3. 避免频繁重启服务器，影响 WAL 效率
4. 定期检查日志文件完整性

### 相关文件
- `default.wal` - Write-Ahead Log 文件
- `default.jdb` - 数据库快照文件
